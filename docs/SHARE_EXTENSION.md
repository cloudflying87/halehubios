# Safari Share Extension — how it stays in the build

"Share → HaleHub" in Safari imports a recipe from the live tab
(`ShareExtension/GetPageContent.js` captures the rendered DOM, then
`ShareViewController.swift` POSTs it to the API using the shared login token).

## ⚠️ Why it disappeared once (and must not again)

The extension target was originally added by hand in Xcode (see
`ShareExtension/SETUP.md`) but was **never defined in `project.yml`**. Because
this project is generated by **XcodeGen** (`xcodegen generate`), every
regeneration rebuilt `HaleHubIOS.xcodeproj` from `project.yml` — and silently
dropped the manually-added target. A TestFlight build archived from a
regenerated project shipped with **no extension**, so "Share → HaleHub" was
missing on fresh installs even though it still worked on machines that had an
older, extension-bearing build.

The same trap applied to the main app's **App Group entitlement**: the
`.entitlements` file existed on disk but was not referenced in `project.yml`, so
it too was dropped on regeneration.

## The fix (now in `project.yml`)

1. **`ShareExtension` target** — `type: app-extension`, sources = `ShareExtension/`
   (excluding `SETUP.md`, `Info.plist`, `ShareExtension.entitlements`), with
   `INFOPLIST_FILE` and `CODE_SIGN_ENTITLEMENTS` pointing at the checked-in files.
2. **Embedded into the app** — the `HaleHubIOS` target lists
   `dependencies: - target: ShareExtension, embed: true`, which creates the
   "Embed App Extensions" copy phase.
3. **App Group on the app** — `CODE_SIGN_ENTITLEMENTS:
   Sources/HaleHubIOS/HaleHubIOS.entitlements` on the app target. Both targets
   carry `group.com.halefamily.halehubios`.
4. **Versions never drift** — `ShareExtension/Info.plist` uses
   `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` so the extension version
   always matches the container app (Apple requires this; a mismatch can fail
   validation). `bump_build.sh` bumps `CURRENT_PROJECT_VERSION`, which both
   targets inherit.

## Rules going forward

- **Never** add the extension target through Xcode's UI — edit `project.yml`,
  then `xcodegen generate`. Anything added in the UI is lost on the next regen.
- The App Group id (`group.com.halefamily.halehubios`) and token key
  (`halehub_access_token`) in `ShareViewController.swift` must stay in sync with
  `AuthManager` / `APIClient`.

## Verify it's really in a build

`verify_project.sh` is now wired in as an **Archive pre-action** on the
`HaleHubIOS` scheme (`schemes.HaleHubIOS.archive.preActions` in `project.yml`),
so it runs automatically every time you do Product → Archive — no separate
manual step. If the Share Extension, its entitlements, or the shared scheme
are missing after a fresh regen, the script exits non-zero and the archive
should stop before it wastes your time.

This was wired up without a Mac in the loop to test it, so **the first time
you archive after pulling this change, confirm it actually ran**: Xcode's
Report Navigator (⌘9) → the archive build → look for a "Verify Project" step
before compilation starts, and make sure a failing check really blocks the
archive rather than just logging a warning. If it doesn't block, treat this
as unverified and keep doing the manual check below until it's confirmed.

Manual check (still useful for a plain debug build, or if the pre-action
turns out not to be enforced):

```sh
xcodegen generate
xcodebuild -project HaleHubIOS.xcodeproj -scheme HaleHubIOS \
  -destination 'generic/platform=iOS' -configuration Release build
# then confirm the extension is embedded:
ls "$(…DerivedData…)/HaleHubIOS.app/PlugIns/"   # must list ShareExtension.appex
```

If `PlugIns/ShareExtension.appex` is present, the archived build will contain the
extension. On the device, the user must open HaleHub and log in once (the
extension reads the shared token), then enable "HaleHub" in Safari's share sheet
(Share → More/Edit Actions) if iOS hasn't surfaced it yet.
